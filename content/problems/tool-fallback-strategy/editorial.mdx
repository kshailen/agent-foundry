## Approach

The research assistant relies on a `primary_search` tool that intermittently throws `ConnectionError` exceptions. When it fails, the agent crashes — even though a perfectly functional `backup_search` tool exists in the codebase but is never registered or used. The user gets no results half the time.

The fix wraps the primary search call in error handling so that failures are caught and returned as a message (not an exception), and registers both tools with the agent. The agent's prompt is updated to enforce a fallback strategy: always try `primary_search` first, and only call `backup_search` if the primary returns an error. In LangGraph, this maps to a conditional edge after the primary search node — route to backup if it failed, otherwise proceed to the response. The key constraint is that the backup should only trigger on actual failure, not on every request.

## Solution

### LangChain

```python
from langchain_openai import ChatOpenAI
from langchain.agents import AgentExecutor, create_tool_calling_agent
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.tools import tool

llm = ChatOpenAI(model="gpt-4o-mini")


@tool
def primary_search(query: str) -> str:
    """Search using the primary search engine. May occasionally be unavailable."""
    import random
    if random.random() < 0.5:
        return f"ERROR: Primary search service is temporarily unavailable for query '{query}'. Try backup_search."
    return f"Primary results for '{query}': Article about {query} from TechNews, Research paper on {query} from ArXiv"


@tool
def backup_search(query: str) -> str:
    """Search using the backup search engine. Use this when primary_search fails."""
    return f"Backup results for '{query}': Blog post about {query}, Wikipedia article on {query}"


prompt = ChatPromptTemplate.from_messages([
    ("system", (
        "You are a research assistant. When searching for information, follow this strategy:\n"
        "1. Always call primary_search first.\n"
        "2. If primary_search returns an error message (contains 'ERROR'), call backup_search with the same query.\n"
        "3. If primary_search succeeds, use those results. Do NOT call backup_search when primary works.\n"
        "4. Never show raw error messages to the user — present search results naturally."
    )),
    ("human", "{input}"),
    ("placeholder", "{agent_scratchpad}"),
])

agent = create_tool_calling_agent(llm, [primary_search, backup_search], prompt)
executor = AgentExecutor(agent=agent, tools=[primary_search, backup_search])

result = executor.invoke({"input": "Find recent research on transformer architectures"})
print(result["output"])
```

## Why This Works

The critical change is converting the `ConnectionError` exception into a returned error string. When a tool raises an exception, many agent frameworks crash or produce unhelpful error traces. By catching the error inside the tool and returning a clearly tagged message (prefixed with "ERROR:"), the agent receives structured feedback it can act on — the LLM sees the error string and knows to invoke the fallback tool.

Registering both tools with the agent and providing clear fallback instructions in the prompt completes the pattern. The agent tries `primary_search` first (per instruction), reads its output, and conditionally calls `backup_search` only when the primary returns an error. This ensures the backup is never called unnecessarily (saving the overhead of a redundant call) while guaranteeing the user always gets results.

## Key Takeaways

- Convert tool exceptions into returned error strings so the agent can react programmatically instead of crashing.
- Register both primary and fallback tools, and use the system prompt to enforce try-primary-then-fallback ordering.
- Tag error returns with a clear prefix (e.g., "ERROR:") so the LLM can reliably distinguish failures from valid results.
