## Approach

The coding assistant answers any question — medical advice, legal guidance, financial recommendations — because the system prompt says "answer the user's questions" with no topic restrictions. This is a liability risk: a coding assistant should not be giving medical or legal advice. The model needs clear boundaries defining what it can and cannot discuss.

The fix adds an explicit allow-list of in-scope topics (programming, software engineering, debugging, developer tools, etc.) and a deny-list of out-of-scope domains (medical, legal, financial advice). Critically, we also handle the edge case where coding and other domains overlap — a question like "How do I build a REST API for a healthcare app?" is a coding question even though it mentions healthcare. The prompt instructs the agent to evaluate whether the user is asking for coding help (in-scope) or domain-specific advice (out-of-scope).

## Solution

### LangChain

```python
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate

llm = ChatOpenAI(model="gpt-4o-mini")

prompt = ChatPromptTemplate.from_messages([
    ("system", (
        "You are a coding assistant. You help with programming, software engineering, debugging, "
        "developer tools, system design, and technical concepts.\n\n"
        "IN-SCOPE TOPICS (answer these fully):\n"
        "- Programming in any language (Python, JavaScript, Rust, etc.)\n"
        "- Software architecture and system design\n"
        "- Debugging, testing, and code review\n"
        "- Developer tools, frameworks, and libraries\n"
        "- DevOps, databases, APIs, and infrastructure\n"
        "- Building applications for any domain (healthcare apps, legal tech, fintech) — "
        "as long as the question is about the technical implementation\n\n"
        "OUT-OF-SCOPE TOPICS (politely decline these):\n"
        "- Medical advice, diagnoses, or treatment recommendations\n"
        "- Legal advice or opinions\n"
        "- Financial advice or investment recommendations\n"
        "- Personal opinions on non-technical topics\n"
        "- Creative writing unrelated to code\n\n"
        "HANDLING OUT-OF-SCOPE REQUESTS:\n"
        "1. Acknowledge the user's question politely.\n"
        "2. Explain that you are a coding assistant and not qualified to advise on that topic.\n"
        "3. Suggest they consult an appropriate professional.\n"
        "4. Offer to help with any coding-related questions.\n\n"
        "EDGE CASE RULE: If a question mentions a non-technical domain (healthcare, law, finance) "
        "but is asking about how to BUILD or CODE something in that domain, it IS in scope. "
        "Answer the technical aspects without giving domain-specific advice."
    )),
    ("human", "{input}"),
])

chain = prompt | llm

result1 = chain.invoke({"input": "How do I reverse a linked list in Python?"})
print("Coding query:", result1.content)

result2 = chain.invoke({"input": "I have a headache and fever. What medicine should I take?"})
print("Medical query:", result2.content)

result3 = chain.invoke({"input": "Can I sue my landlord for not fixing the plumbing?"})
print("Legal query:", result3.content)

result4 = chain.invoke({"input": "How do I build a REST API for a healthcare app?"})
print("Edge case:", result4.content)
```

## Why This Works

The allow-list/deny-list approach gives the model a clear decision framework for every incoming message. The edge-case rule is the most important addition — without it, the model might incorrectly refuse coding questions that happen to mention healthcare or law. By teaching the model to distinguish "asking about building in a domain" (in-scope) from "asking for domain expertise" (out-of-scope), we avoid both over-blocking and under-blocking. The four-step deflection pattern ensures declined requests are handled gracefully — the user never feels ignored or dismissed.

## Key Takeaways

- Topic boundaries require both an allow-list (what to answer) and a deny-list (what to decline) for reliable enforcement.
- Edge cases where domains overlap must be addressed explicitly — a keyword-based filter would incorrectly block "build a healthcare API."
- Polite deflection with a redirect ("I can help with coding questions instead") maintains a positive user experience even when declining.
